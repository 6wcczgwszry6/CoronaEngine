# F5 运行复盘记录

本文档用于记录每一次 F5 / 实机运行后的证据、结论与改进项。原则是：以日志和实际界面观察为准，区分已确认问题、用户观察、待验证假设，避免把一次运行中的外部服务失败误判成核心架构失败。

## 记录规范

每次运行后追加一条记录，至少包含：

- 运行时间与日志文件
- 本次验证目标
- 关键现象
- 已确认问题
- 待验证假设
- 下一步改进项
- 是否阻断终极计划验收

问题优先级建议：

- P0：阻断模型对话、Seed Plan、批次生成、动态介入主链路
- P1：不阻断主链路，但明显影响演示稳定性、交互理解或用户信心
- P2：体验、可观测性、边缘能力或后续工程质量改进

### 已确认问题

#### P0：确认 GM 提案后没有进入 Seed Plan / 执行闭环

证据等级：[用户 transcript 暴露的实际问题] / [待日志补证]

现象：

- 房主两次确认 GM 提案后，只得到“已确认 gm-...”。
- transcript 中没有出现类似“方案已确认”“开始生成”“第 1 批生成中”“资源调度中”“Seed Plan confirmed/executing”的用户可见反馈。
- 房主不得不追问“为什么不执行呀”。

判断：

当前主链路很可能卡在 GM conflict proposal confirmation 与 Seed Plan / GenerationScheduler submit 之间。也就是说，系统已经能做“提案-确认”的表层交互，但没有把确认语义稳定转成“构造 Seed Plan 并进入执行”的控制面动作。

可能根因：

- GM 提案类型被当成通用冲突提案，而不是“可执行 Seed Plan 草案”。
- `@小女孩 做一个...` 这种面向 Agent 的执行请求被 GM 仲裁层截获，生成了冲突确认，而没有进入 Seed Plan builder。
- 确认 `gm-...` 后只更新 proposal 状态，没有调用 plan confirmation / scheduler submit。
- 当前确认动作缺少明确 `proposal_type` / `intent_type`，导致后端无法区分“讨论冲突确认”和“执行方案确认”。

影响：

- 直接阻断终极计划中的“最终方案确认 -> 场景生成”。
- 多人/多 Agent 讨论能够启动，但无法自然过渡到批次生成。
- 用户体验上会表现为“系统一直让确认，但确认完不干活”。

下一步改进：

- GM proposal 必须区分 `clarification_proposal`、`conflict_resolution`、`seed_plan_proposal`、`execution_confirmation`。
- 对“明确生成场景”的房主请求，应优先生成 Seed Plan 草案，而不是泛化成冲突提案。
- 房主确认 `seed_plan_proposal` 后，必须进入 `SeedPlan.confirmed -> scheduler submit -> stage=资源调度/生成中`。
- 如果当前能力暂不执行真实生成，UI 必须明确说明“已确认方案，但真实生成 provider 未启用/待执行”，不能静默。

#### P0：执行意图被错误包装为“可能影响多人共识”的通用冲突提案

证据等级：[用户 transcript 暴露的实际问题]

关键表现：

- 对“夜晚幻想集市，有入口、摊位、灯光、小休息区，整体风格统一”这种完整生成意图，GM 输出的是：
  - “潜在冲突：暂无明确对象冲突，但该操作可能影响多人共识或核心布局。”
  - “建议：先保留最近用户明确操作，Agent 物体让位...”
- 这套话术适合介入已有生成或多人冲突处理，不适合首次执行方案确认。

判断：

当前 GM 仲裁策略过宽，把普通生成请求也套进“冲突/覆盖多人意见”的流程，导致执行入口被确认流程吞掉。对单人或房主明确主导的场景，这会制造不必要的确认负担。

下一步改进：

- 增加 intent gate：
  - 初始阶段 + 房主明确生成目标 -> `seed_plan_proposal`
  - 已有执行中 + 修改/删除/覆盖/冲突 -> `intervention_or_conflict`
  - 非房主含糊建议 -> `discussion_note`
- 单人/房主主导模式下，初始生成不应触发“影响多人共识”式冲突确认，除非确有其他用户意见冲突。

- F5 复盘中用户问 `@商人 查看生成情况`、`@商人 到哪步了`、`@GM 为什么不执行` 时，系统容易先记录为可介入窗口，再继续调用角色 Agent 或生成 GM 提案。
- 根因是 `InteractionCoordinator` 已经具备 `status_query` 只读判断，但 `LANChatAgentWorker` 只把普通聊天室消息同步给 Coordinator；对 `@Agent` trigger，状态查询同步后仍会继续进入 busy/runtime、角色模型或 GM proposal 路径。
- 这会造成两个问题：一是用户查询进度时得到“我会在下一阶段吸收”这种误导回复；二是查询类消息可能污染 pending intervention / proposal / SeedPlan 队列。

### 已落地

- `LANChatAgentWorker._process_trigger()` 在 GM pace / clarification 之后、角色 Agent 调用之前，新增 Coordinator `status_query` 短路。
- `@Agent 查看生成情况 / 到哪步了 / 生成到哪里了 / 为什么不执行` 现在会：
  - 调用 `InteractionCoordinator.ingest_message()`；
  - 只接收 `event_type == "status_query"` 的只读结果；
  - 直接向聊天室返回当前方案状态；
  - 不调用角色模型；
  - 不生成 GM proposal；
  - 不进入 host executor。
- 这个修复只影响查询类 trigger；真实生成、修改、删除、追加仍走原有 Coordinator / runtime / host executor 路径。

### 已自动验证

- `python editor/plugins/AITool/services/test_lanchat_agent_orchestrator.py`
- `python -m py_compile editor/plugins/AITool/services/lanchat_agent_worker.py editor/plugins/AITool/services/test_lanchat_agent_orchestrator.py`

新增测试覆盖：

- 有 active SeedPlan 时，`@商人 查看生成情况，到哪步了` 返回 Coordinator 状态。
- 状态查询不会调用模型 Agent。
- 状态查询不会进入 host executor。
- 状态查询不会产生 GM proposal。

### 待 F5/实机验证

- 生成中询问 `@商人 查看生成情况 / 到哪步了` 时，应只返回当前队列、生成或批次状态。
- 询问 `@GM 为什么不执行` 时，应解释排队/资源调度/当前批次状态，不应创建新的 SeedPlan 或 GM 提案。
- 聊天室不应再把查询类消息显示成“可介入窗口：你的补充会优先进入下一批”。

## 2026-06-18 追加修复：GM 确认/拒绝后前端按钮立即失效

### 已定位

- F5 复盘中出现过：房主拒绝某个 `gm-...` 提案后，消息气泡下方仍保留“确认 / 拒绝”按钮；再次点击时，后端只能返回“已处理 rejected，不会重复入队”。
- 后端幂等保护是对的，但前端继续展示按钮会让用户误以为该提案仍可操作，降低演示可信度。
- 根因是前端只在点击后调用 `dismissDisclosureByProposal()` 清掉顶部 disclosure，没有维护“已处理 proposal id”，也没有让消息气泡下方的 `gm_proposal` 按钮读取处理状态。

### 已落地

- `lanchat.js` 增加 `processedProposalIds`，并提供：
  - `markProposalHandled(proposalId)`
  - `isProposalHandled(proposalId)`
- 点击 GM 确认/拒绝后，RoomPanel 会立即调用 `markProposalHandled()`，消息气泡下方按钮即时隐藏。
- 收到 GM 回执消息时，store 会从“已确认 / 已取消 / 已拒绝 / 已处理 / 不会重复 / 不会执行”等文本中提取 `gm-* / fa-* / cr-*`，同步标记为已处理，覆盖用户手动输入 `@GM 确认 ...` 的路径。
- 顶部 disclosure 的 pending 优先级也改为只保留“未处理”的确认项；已处理提案不会继续压住常规进度状态。

### 待 F5/实机验证

- 点击某个 GM 提案的“确认”或“拒绝”后，该消息下方按钮应立即消失。

## 2026-06-18 追加修复：GenerationScheduler 生命周期日志可对账

### 已定位

- F5 复盘中多次出现 UI 显示 `queued / 生成中 0%`，但 `corona.log` 里难以按 `seed_id / generation_id / batch_id` 追踪任务到底卡在 submit、queue、worker、provider 还是 import。
- `GenerationScheduler` 已经维护 `recent_events`，但主要在内存 snapshot 里，不保证每个状态变化都落到日志。
- 缺少落盘日志会让后续 F5 一旦卡住，很难判断是 provider 慢、scheduler 暂停、队列拥堵、composer factory 缺失，还是 UI 映射错误。

  - `event`
  - `room_id`
  - `session_id`
  - `plan_id`
  - `job_id`
  - `batch_id`
  - `status`
  - `stage`
  - `priority`
  - `cancelled_count / pruned_count`
- 日志只使用调度器已有白名单事件字段，不写入：
  - `prompt`
  - `provider`
  - `runtime_context`
  - `token / api_key`
  - 任何模型请求正文
- 覆盖的关键事件包括：
  - `submit`
  - `queue_full`
  - `status_change`
  - `cancel_requested`

## 2026-06-18 追加修复：host-only disclosure fallback 可被房主端还原

### 已定位

- F5 复盘里多次出现系统消息只显示“有一项仅房主可见的进度更新。”或“有一项需要房主确认的事项。”。
- 这是因为 native/host targeted API 不可用时，`LANChatAgentWorker` 为避免泄漏房主专属信息，会把 host disclosure 降级成 participant-safe metadata。
- 降级策略本身是对的，但前端房主端也只能看到 participant-safe 版本，导致房主无法在顶部状态区看到真实确认事项、仲裁建议或进度内容。

### 已落地

- `LANChatAgentWorker._emit_new_disclosure_events()` 在 targeted host API 不可用时，现在会在同一条广播 metadata 中同时写入：
  - `disclosure`：参与者安全视图，只保留泛化文案和安全 proposal id。
  - `host_disclosure`：房主可恢复视图，保留房主应看到的确认文案、可操作按钮、proposal id 和安全元数据。
- `lanchatDisclosure.extractDisclosureFromMessage()` 增加 `role` 参数：
  - `role === "host"` 时优先解析 `metadata.host_disclosure`。
  - 非房主继续解析 `metadata.disclosure`，不会看到房主专属文案。
- `lanchat.js` 在从消息生成 disclosure card 时传入 `state.role`，避免房主端退化为普通参与者视图。
- 静态约束脚本增加检查，防止后续改动遗漏 role-aware fallback。

- `node editor/Frontend/src/stores/lanchatDisclosure.test.mjs`
- `node editor/Frontend/scripts/test-lanchat-roster.mjs`
- `python editor/plugins/AITool/services/test_lanchat_agent_orchestrator.py`
- `python -m py_compile editor/plugins/AITool/services/lanchat_agent_worker.py editor/plugins/AITool/services/test_lanchat_agent_orchestrator.py`

新增测试覆盖：

- 同一条 host-only fallback 广播中，普通成员只能解析到安全占位文案。
- 房主能从 `host_disclosure` 还原真实确认文案、可操作动作和 proposal id。
- disclosure metadata 仍过滤 `prompt/provider/raw_prompt/hidden_debug_ref` 等内部字段。
- 后端广播文本仍是安全占位，不在公开聊天气泡中泄漏房主专属内容。

### 待 F5/实机验证

- 房主端确认按钮应能基于 `host_disclosure.available_actions` 正常显示并执行。
- 聊天公开气泡中不应出现 prompt、provider、worker、queue、batch id 等内部链路信息。
